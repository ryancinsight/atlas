## Per-file dependency ordering

The 6 named files have non-trivial order dependencies so the inner commit can land as a single atomic commit (ADR 0011 §Decision §Leg 1 §Class D file-rite category: atomic commit per task on inner-helios):

1. **`Cargo.toml`** — root cause for the dep-edge removals (H-061 + H-062 feature edges). Must land before `Cargo.lock`.
2. **`Cargo.lock`** — regenerated by the Cargo workspace after `Cargo.toml`'s dep-edge removals. Depends on `Cargo.toml` landing first.
3. **`gap_audit.md`** — H-061/H-062 entry adds; should land before `CHANGELOG.md` because the CHANGELOG narrative refers back to the gap-closure trail.
4. **`CHECKLIST.md`** — H-061 + H-062 done-marking; precedes `backlog.md` (the checklist is upstream of the backlog).
5. **`backlog.md`** — H-061 + H-062 task-row past-tense collapses; precedes `CHANGELOG.md` (the backlog is upstream of the changelog's task reference).
6. **`CHANGELOG.md`** — `## Unreleased Sprint-1 pre-session WIP cleanup` heading + summary narrative; depends on `gap_audit.md` + `CHECKLIST.md` + `backlog.md` (the changelog aggregates their trail).

The inner commit can land atomically with all 6 files stage-added in one command (the staging order is irrelevant; `git commit` writes the tree atomically), but the per-file EDITING should follow the order above so the diff narrative reads naturally.

## Failure modes / risks

- **23 unnamed count drift.** The §D SSOT claimed 23 unnamed additional dirty entries, but the actual current `git -C repos/helios status --short | wc -l` may differ (pre-commit gate re-validates). If the actual count is smaller (some peer landed those entries as un-pushed commits), the inner commit absorbs only the remaining; the PR description's per-class breakdown (estimates ~6/5/5/7 across the 4 sub-trees) will need adjustment.
- **H-061/H-062 ordering risk.** If `Cargo.toml` lands without `Cargo.lock` regenerated, downstream CI fails the `--locked` gate. Mitigation: `cargo build --all-features --locked` pre-commit gate is mandatory; the dependency-ordering section above enforces the Cargo.toml-→-Cargo.lock sequence.
- **PR 0007 cross-PR contamination.** Bundling 29 files into a single inner commit conflates otherwise-independent task boundaries (H-061 + H-062 feature-edge removals are technically distinct from Sprint-1 task captures and §D-denormalization). Mitigation: per-file dependency ordering maintains task boundaries where they exist; the H-061/H-062 work groups naturally with the `Cargo.toml` + `Cargo.lock` edits, while Sprint-1 task captures group with `gap_audit.md` + `CHECKLIST.md` + `backlog.md` + `CHANGELOG.md`. The inner commit's body enumerates them; if more granular inner commits are preferred (one per task boundary), split into PR 0007a (H-061/H-062 dep-edge strip) + PR 0007b (Sprint-1 task captures) — but the bundled 6-file commit is the recommended default for this cleanup event.
- **Target-branch choice.** Target branch is TBD by peer-session owner. If the target branch is NOT the atlas-meta-pinned branch, the inner commit appears as a new commit in atlas-meta's gitlink (submodule-pointer drift). Mitigation: pre-commit gate `git -C repos/helios rev-parse --abbrev-ref HEAD` confirms target branch; if it differs from the atlas-meta-pinned branch, peer-session owner chooses either (a) `git rebase` onto the atlas-meta-pinned branch, or (b) accept submodule pointer drift + a follow-up `chore(atlas): Advance helios submodule pointer` commit.
- **`internal-dirty-batch1` naming drift.** A future codex session could mis-file a cross-Atlas migration batch under the `internal-dirty-batchN` sub-counter (it should be `batchN`). Mitigation: the §Per-batch reservations table in `D:/atlas/checklist.md` is the SSOT for batch reservations; new reservation rows go through checklist.md updates + ADR 0010 cross-walk. The `internal-dirty-batchN` sub-counter is reserved exclusively for inner-cleanup events per this PR's documented intent.
- **Cross-repo tag namespace pollution.** Multiple submodules could adopt the `internal-dirty-batch1` sub-counter once it's introduced. Mitigation: ADR 0010's cross-repo tag namespace collision rule applies — each submodule's tag is prefixed with `{consumer-repo}/`, so `helios/internal-dirty-batch1` and `apollo/internal-dirty-batch1` coexist cleanly.

## Sequencing (this PR's inner-helios atomic ceremony)

1. **Inner commit**: `git -C repos/helios commit -m 'refactor(helios): Sprint-1 pre-session WIP cleanup ...'` — single atomic commit absorbing 6 named + 23 unnamed = 29-file diff.
2. **Inner tag**: `git -C repos/helios tag -a helios/atlas-migration-push/internal-dirty-batch1 <inner-SHA> -m '...see tag annotation script above...'`.
3. **Inner push**: `git -C repos/helios push origin <target-branch> --tags`.
4. **(Optional) Atlas-meta pointer advance**: only if the inner commit lands on a branch not already gitlinked at atlas-meta — `git -C D:/atlas add repos/helios && git -C D:/atlas commit -m 'chore(atlas): Advance helios submodule pointer to <inner-SHA>'`.
5. **(Optional) Atlas-meta docs-rounding**: only if the pointer-advance happened — `git -C D:/atlas add backlog.md checklist.md gap_audit.md && git -C D:/atlas commit -m 'chore(atlas): Sync helios pointer <inner-SHA> + inner-dirty cleanup record'`.

Steps 1-3 are mandatory. Steps 4-5 conditional on the test in step 1's pre-commit gate (`git -C repos/helios rev-parse --abbrev-ref HEAD`).

## Out of scope

- **`repos/helios/<src-tree>` per-task dirty entries** — the §D-23 unnamed aggregate ENVELOPE is in scope, but each per-task dirty entry's individual resolution (e.g., H-033 gEUD closure) is its own concern; they get bundled into the cleanup event but their semantic closure is recorded separately in `repos/helios/CHANGELOG.md` per inner-submodule convention.
- **Atlas-parent file-system changes** — per ADR 0011 §Decision §Leg 2 disjoint-scope rule; **NO ATLAS-META RECLAIM** for any of the §D-internal directories. Atlas-meta only touches its own PM artifacts (this docs/pr/ file, possibly the §D table SSOT retraction on next pass).
- **Migration batches #1-#6** — those are separate per ADR 0010 reservations (`kwavers/atlas-migration-push/batch1`, `cfdrs/atlas-migration-push/batch2`, `ritk/atlas-migration-push/batch3`, `kwavers/atlas-migration-push/batch4`, `apollo/atlas-migration-push/batch5`, `cfd-core+ritk-core+moirai/atlas-migration-push/batch6`); this PR's `internal-dirty-batch1` is a sub-counter for inner-cleanup events and is NOT a substitute for any migration-batch tag.
- **A separate ADR for the `internal-dirty-batchN` sub-counter.** The sub-counter convention is documented here (this PR doc) + cross-referenced from checklist.md §Per-batch reservations (next pass). Authors may promote it to ADR 0012+ if the sub-counter becomes more widely adopted across submodules.
